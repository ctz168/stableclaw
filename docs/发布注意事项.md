# stableclaw npm 发布注意事项

> 本文档记录了手动发布 stableclaw 到 npm 时必须遵守的流程和踩过的坑。
> 违反其中任何一条都可能导致发布的包在生产环境无法运行。

## 一、发布前置条件

| 条件 | 要求 |
|------|------|
| Node.js | >= 22.12 |
| pnpm | 必须在 PATH 中可调用（含 corepack shims 环境） |
| npm token | 需要有 `stableclaw` 包的发布权限（Automation 类型） |
| 分支 | 建议从 `main` 分支发布 |

## 二、发布流程（必须按顺序执行）

### 步骤 1：Bump 版本号

编辑 `package.json`，修改 `version` 字段：

```bash
# 项目使用 calver 日期版本号格式：YYYY.M.MINOR
# 例如 2026.4.9 → 2026.4.10
```

同时运行 `node scripts/build-stamp.mjs` 同步构建时间戳。

### 步骤 2：安装依赖

```bash
pnpm install
```

### 步骤 3：执行完整构建（最关键的一步）

**必须依次执行两条命令，缺一不可：**

```bash
pnpm build      # 核心 TypeScript 编译
pnpm ui:build   # Dashboard 控制面板 UI 构建
```

> ⛔ **注意：`pnpm build` 不包含 `pnpm ui:build`！它们是两条独立的命令。**
> 如果只跑 `pnpm build` 而不跑 `pnpm ui:build`，发布的包打开 Dashboard 时会报错：
> `Control UI assets not found.`

**绝对不能跳过任何一条。** `pnpm build` 会调用 tsdown 编译整个项目，生成以下关键文件：

- `dist/entry.js` — CLI 入口文件
- `dist/index.js` — 主模块
- `dist/extensions/*/runtime-api.js` — 各扩展的运行时接口
- `dist/plugin-sdk/*.js` — 插件 SDK 导出
- `dist/cli-startup-metadata.json` — CLI 启动元数据

`pnpm ui:build` 会生成 Dashboard 控制面板所需的前端资源：

- `dist/control-ui/index.html` — Dashboard 入口页面
- `dist/control-ui/assets/*.js` — Dashboard 前端 JS 资源
- `dist/control-ui/assets/*.css` — Dashboard 样式资源

> ⚠️ **踩坑记录 1（2026.4.8 / 2026.4.9）**：跳过了 `pnpm build`，导致 `dist/entry.js` 不存在。
> 用户安装后运行报错：`Error: stableclaw: missing dist/entry.(m)js (build output).`
>
> ⚠️ **踩坑记录 2（2026.4.10）**：跑了 `pnpm build` 但没跑 `pnpm ui:build`，
> 导致 `dist/control-ui/` 目录缺失。用户安装后打开 Dashboard 报错：
> `Control UI assets not found. Build them with pnpm ui:build.`
> **这两条命令都必须跑！**

### 步骤 4：验证构建产物

构建完成后，**必须确认以下文件全部存在**（任何一项缺失都说明构建不完整，不能发布）：

```bash
# === 核心（缺一不可，否则启动直接崩溃）===
ls dist/entry.js
ls dist/index.js

# === Dashboard UI（缺失则打开 Dashboard 报错）===
ls dist/control-ui/index.html

# === 扩展运行时接口 ===
ls dist/extensions/speech-core/runtime-api.js
ls dist/extensions/image-generation-core/runtime-api.js
ls dist/extensions/media-understanding-core/runtime-api.js

# === 插件 SDK 导出 ===
ls dist/plugin-sdk/core.js
ls dist/plugin-sdk/compat.js
```

**特别注意**：如果 `dist/control-ui/index.html` 不存在，说明忘了跑 `pnpm ui:build`，
必须补跑后再发布。

也可以运行项目自带的发布检查脚本（会自动检查上述文件）：

```bash
node --import tsx scripts/release-check.ts
```

这个脚本会自动检查包内容、版本号、扩展清单、包大小等。

### 步骤 5：Dry-run 打包测试

```bash
npm pack --dry-run
```

确认打包过程没有报错，且输出中包含 `dist/entry.js`。

### 步骤 6：配置 npm 认证并发布

```bash
# 写入 token 到 .npmrc（发布后会自动清理）
echo "//registry.npmjs.org/:_authToken=<你的npm_token>" >> .npmrc

# 发布
npm publish --access public
```

或者使用一键发布脚本：

```bash
./scripts/publish-npm.sh <npm_token>
```

### 步骤 7：发布后验证

```bash
# 确认 registry 上版本正确
npm view stableclaw version

# 全局安装测试
npm install -g stableclaw@latest
stableclaw --version
```

### 步骤 8：推送代码到 Git

```bash
git add package.json
git commit -m "chore: bump version to <新版本号>"
git push origin main
```

## 三、踩坑记录

### 坑 1：漏跑 UI 构建（2026.4.10）

**现象**：用户 `npm install -g stableclaw@latest` 后，打开 Dashboard 报错：
```
Control UI assets not found. Build them with `pnpm ui:build` (auto-installs UI deps),
or run `pnpm ui:dev` during development.
```

**原因**：`pnpm build` 和 `pnpm ui:build` 是两条**独立**的命令。
`pnpm build` 只负责 TypeScript 编译（tsdown），不会构建 Dashboard 前端资源。
如果发布前只跑了 `pnpm build` 而没跑 `pnpm ui:build`，
包里就不会有 `dist/control-ui/` 目录，Dashboard 无法加载。

**教训**：发布前必须跑**两条**命令：
```bash
pnpm build      # 后端/核心编译
pnpm ui:build   # Dashboard 前端构建
```
漏掉任何一条都会导致用户端功能异常。

### 坑 2：未构建直接发布（2026.4.8 / 2026.4.9）

**现象**：用户 `npm install -g stableclaw@latest` 后，运行 `stableclaw` 命令报错：
```
Error: stableclaw: missing dist/entry.(m)js (build output).
```

**原因**：发布前没有执行 `pnpm build`，`dist/` 目录中缺少 `entry.js`。
`stableclaw.mjs`（CLI 启动脚本）在第 173-178 行会尝试导入 `./dist/entry.js`，
找不到时抛出这个致命错误，整个程序无法启动。

**教训**：`dist/` 不在 git 版本控制中，它只存在于本地构建后。发布时 npm 打包的是
本地文件系统中的文件，不会自动帮你构建。**永远先构建再发布。**

### 坑 3：pnpm 不在 PATH 中

**现象**：`pnpm build` 命令报错 `Command "build" not found`，或 `scripts/tsdown-build.mjs`
执行后 `dist/` 目录没有任何变化。

**原因**：在某些环境中（如仅有 corepack 没有 pnpm 全局安装），
`pnpm` 命令不在 PATH 里。构建脚本内部调用 `pnpm exec tsdown` 时静默失败。

**解决**：确保 `pnpm` 可用：
```bash
# 方法 1：corepack 自动安装
corepack enable

# 方法 2：npm 全局安装
npm install -g pnpm

# 方法 3：手动加入 corepack shims 到 PATH
export PATH="/usr/lib/node_modules/corepack/shims:$PATH"
```

### 坑 4：旧 dist 残留混淆

**现象**：构建后发布的包运行异常，某些文件是旧版本的。

**原因**：上一次构建的 `dist/` 文件残留，和新构建混合在一起。

**解决**：构建前先清理旧产物：
```bash
rm -rf dist
pnpm build
```

### 坑 5：版本号冲突

**现象**：`npm publish` 报错 `403 Forbidden` 或 `version already published`。

**原因**：`package.json` 中的版本号在 npm registry 上已经存在。

**解决**：先查看已有版本，再 bump：
```bash
npm view stableclaw versions --json
# 修改 package.json 中的 version 为更高版本
```

## 四、一键发布脚本

项目提供了 `scripts/publish-npm.sh`，集成了以上所有检查步骤：

```bash
# 标准用法（自动：检查环境 → 构建 → 验证产物 → 发布）
./scripts/publish-npm.sh <npm_token>

# 通过环境变量传入 token
NPM_TOKEN=npm_xxxxx ./scripts/publish-npm.sh

# 跳过构建（仅当 dist/ 已存在时使用）
./scripts/publish-npm.sh --skip-build <npm_token>
```

脚本内置了以下安全机制：
- **构建产物验证**：检查 `dist/entry.js`、`dist/control-ui/index.html`、`dist/index.js` 等关键文件
- **npm pack 预检**：发布前执行 dry-run 确保包能正常打包
- **corepack shim 回退**：自动检测 pnpm 可用性
- **旧 dist 清理**：构建前自动清理旧的 `dist/` 目录

## 五、构建命令详情

`pnpm build` 实际执行的是一条很长的构建链：

```
pnpm canvas:a2ui:bundle           # Canvas A2UI 资源打包
node scripts/tsdown-build.mjs     # 核心 TypeScript 编译（tsdown）
node scripts/runtime-postbuild.mjs # 运行时后处理
node scripts/build-stamp.mjs      # 写入构建时间戳
pnpm build:plugin-sdk:dts         # 插件 SDK 类型声明
node scripts/write-plugin-sdk-entry-dts.ts
node scripts/check-plugin-sdk-exports.mjs
node scripts/canvas-a2ui-copy.ts
node scripts/copy-hook-metadata.ts
node scripts/copy-export-html-templates.ts
node scripts/write-build-info.ts
node scripts/write-cli-startup-metadata.ts  # CLI 启动元数据
node scripts/write-cli-compat.ts
```

其中最核心的是 `tsdown-build.mjs`，它负责编译 `src/entry.ts` → `dist/entry.js`。

**但注意：以上构建链不包含 Dashboard UI！** UI 构建是独立的：
```
pnpm ui:build   # 等同于 node scripts/ui.js build
```

如果手动分步构建，至少要执行：
```bash
# 后端核心（pnpm build 的子集）
node scripts/tsdown-build.mjs
node scripts/runtime-postbuild.mjs
node scripts/build-stamp.mjs

# Dashboard UI（必须单独执行！）
pnpm ui:build
```
